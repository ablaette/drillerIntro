PolMine-Framework und R-Paket 'driller'

Ausgangslage - Problemstellung - Zielsetzung

  • Für die Korpusanalyse gibt es eine Reihe von Anwendungen:
    • erste Generation (WordSmith, Lexico3)
    • Server-basierte (CQPweb, TXM)
  • Diese Anwendungen sind geschlossene Systeme, die der Methodenentwicklung noch zu viele Grenzen setzen. Für die Erprobung neuer Verfahren führt bislang kein Weg an der Nutzung von Programmiersprachen vorbei. Gut entwickelte Bibliotheken und Tools sind u.a.:
    • CQP
    • NLTK
    • tm, lsa, topicmodels
  • Eine produktive Arbeit mit Korpora sollte einen interaktiven Analysprozess auch mit textstatistischen Verfahren ermöglichen und durchgängig den Weg zurück zum Text offen halten. Weil dies die existierenden Tools nicht leisten können, verbindet das PolMine-Framework für Korpusanalyse existierende Tools und bündelt diese in einem R-Paket 'driller'

Design des PolMine-Frameworks

Architektur

Im Kern verbindet das Framework die Corpus Workbench (CWB) als System für die Korpusverwaltung mit der Nutzung von R für die Datenauswertung. Für die Korpusaufbereitung wird Python genutzt.

Zielsetzungen

  • Portabilität:
  • Performanz
  • Nutzerfreundlichkeit ("usability")
  • Interaktvität

Wie kann ich den 'driller' updaten?

  • Der 'driller' wird über GitHub zur Verfügung gestellt. Die jeweils neueste Version kann in R über das Paket 'devtools' installiert werden. Wenn dieses noch nicht installiert ist:
install.packages("devtools")
  • Um die neueste 'driller'-Version zu installieren, muss man zunächst das 'devtools'-Package laden. Dann kann das Paket von GitHub geladen und installiert werden:
library(devtools)
install_github("ablaette/driller")

Laden von Korpora

Partitionen: Grundlagen

  • Fast jede Funktion des Pakets setzt die Definition eines Subkorpus bzw. eine 'Partition' voraus.

  • Für die Definition von Partitionen werden die Metadaten genutzt, die in CWB-Korpora in den sog. s-Attributen (strukturellen Attributen) gespeichert werden. Diese werden an die Funktion 'partition' mit einer Liste übergeben. So lässt sich beispielsweise eine Partition der Reden im Deutschen Bundestag des Jahres 2010 anlegen:

bt2010 <- partition(corpus="PLPRTXT", def=list(text_year="2010", text_type="speech"))
  • Die Liste kann die Definition beliebig vieler s-Attribute enthalten. Welche s-Attribute es in einem Korpus gibt, kann man mit der Hilfsfunktion 'sAttributes' ermitteln:
sAttributes("PLPRTXT")

Partitionen: Flexibilität

  • In der Liste, die eine Partition definiert, können auch mehrere Werte übergeben werden. Um etwa Korpora von Regierungs- und Oppositionsparteien im 17. Bundestag (2009-2013) zu erstellen:
bt17gov <- partition("PLPRTXT", def=list(text_lp="17", text_party=c("CDU_CSU", "FDP")))
bt17opp <- partition("PLPRTXT", def=list(text_lp="17", text_party=c("SPD", "DIE_LINKE", "B90_DIE_GRUENEN")))
  • Die Voreinstellung der 'partition'-Funktion ist, dass die übergebenen Werte den s-Attributen exakt entsprechen müssen, setzt man den Parameter 'method' auf 'grep', ist die Nutzung regulärer Ausdrücke möglich. Um etwa eine Partition mit den Reden eines Monats anzulegen:
bt2010jan <- partition("PLPRTXT", def=list(text_date="2010-01-.*"), method="grep")
  • Um zu ermitteln, welche Daten in einem Korpus in einem definierten Zeitraum auftreten, kann man die Hilfsfunktion datesPeriod nutzen. Diese Werte können dann an die 'partition' Funktion übergeben werden:
postFukushima <- datesRange("PLPRTXT", c("", ""))
bt2011postFukushima <- partition("PLPRTXT", def=list(text_date=postFukushima))
